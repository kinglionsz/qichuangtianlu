  1. 响应式 Canvas 支持

  当前问题本质：Canvas 内部使用硬编码坐标（1100×770），当 CSS
  缩放适配移动端时，内部绘制的 HUD、海拔图等元素不会自适应缩放。

  技术难点：
  - 整个 trajectory.js 中有大量硬编码坐标（hx, hy, cx, cy 等）
  - 需要重构所有绘制函数支持动态坐标
  - 轨迹点数据也是基于 1100×770 的 Canvas 坐标，需要整体缩放

  方案评估：

  ┌───────────────────┬────────────────────────────────┬────────────────────┐
  │       方案        │              优点              │        缺点        │
  ├───────────────────┼────────────────────────────────┼────────────────────┤
  │ A. 动态计算坐标   │ 改动较小，可以复用现有绘制逻辑 │ 需要维护多套配置   │
  ├───────────────────┼────────────────────────────────┼────────────────────┤
  │ B. 离屏 Canvas +  │ 高性能，兼容 Retian 屏         │ 实现复杂，需要双   │
  │ DPR 缩放          │                                │ Canvas             │
  └───────────────────┴────────────────────────────────┴────────────────────┘

  2. GPX 数据动态导入

  当前问题本质：数据分散在多处，有冗余。GPS 坐标在
  waypoints、checkpoints、challengePoints 中重复存储。

  现状：
  - 项目已经有 trajectoryData.js（手动维护的 Canvas 坐标数据）
  - 还有 trajectoryData-gpx.js（从 1041365gpx.txt 导入，但无海拔数据）
  - 当前使用的是 trajectoryData.js，数据来自 995778gpx.txt

  方案评估：
  - 这个工作本质上是数据格式规范化，风险低
  - 但当前数据已经可用，重构收益有限

  2. GPX 数据动态导入 - 当前数据已稳定运行，这个属于"代码优化"而非"功能缺陷"
