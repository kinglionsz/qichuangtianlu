## 二、问题核实

### 数据源对比

**route-995778.json（原始路书）** - 正确数据：
| 虐点 | km | elev | 名称 |
|------|-----|------|------|
| 1 | 33.43 | 55 | 富民路 |

| 4 | 121.80 | 265 | 径心水库返 |

**trajectoryData.json.js（修复前）** - 错误数据：
| 虐点 | km | elev | 名称 |
|------|-----|------|------|
| 1 | 21.65 | 185 | 虐点1 |
| 2 | 39.64 | 194 | 虐点2 |

| 4 | 53.94 | 194 | 虐点4 |

### 根本原因
1. `challengePoints`使用了错误的硬编码数据（6个虐点），而非原始路书的4个虐点
2. `ELEVATION_BASE_POINTS`也使用了错误的虐点海拔数据进行插值
3. 预计算的`elevationPoints`基于错误的插值数据生成

### 修复1: challengePoints从原始路书导入
```javascript
// 修复前（错误）
export const challengePoints = [
  { km: 21.65, name: '虐点1', elev: 185 },
  { km: 39.64, name: '虐点2', elev: 194 },
  // ...6个错误的虐点
].map(cp => { ... });

// 修复后（正确）
export const challengePoints = routeData.challengePoints.map(cp => {
  const nearest = findNearestPoint(cp.km);
  return { ...cp, x: nearest.x, y: nearest.y };
});
```

## 六、经验总结

### 关键教训
1. **数据来源一致性至关重要** - 所有数据应从单一源（route-995778.json）导入
2. **硬编码数据容易出错** - 应避免多处维护相同数据
3. **预计算数据需与源数据同步** - 修改源数据后必须重新生成预计算数据

### 最佳实践
- 优先从原始数据文件导入，而非硬编码
- 数据修改后需验证插值/预计算结果
- 构建验证是必要的检查步骤
